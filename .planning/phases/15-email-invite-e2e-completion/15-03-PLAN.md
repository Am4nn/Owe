# Plan 15-03: Deep Linking from Email + Web Landing Page

## Goal

Make the email invite CTA button actually route users into the Owe app (or web app) with a meaningful destination — the pending invites screen.

**What exists:**
- Email CTA links to `https://you.owe.amanarya.com` — a dead URL
- App scheme `owe://` configured in `app.json`
- `useNotificationDeepLink` exists for push notification deep links
- Expo Router handles route-based navigation

**What's missing:**
- Email CTA doesn't deep-link into the app
- No web landing page at `you.owe.amanarya.com`
- No deep link handler for invite-specific routes

---

## Tasks

<task id="15-03-A" wave="2">
### A. Update email template CTA links

Modify: `supabase/functions/send-invite-email/index.ts`

Update the CTA button href to use a smart link that:
1. On Android: tries to open `owe://invites` (deep link into app)
2. Falls back to `https://you.owe.amanarya.com?invite=true` (web)

```typescript
// Updated CTA href — use web URL with intent fallback
const ctaUrl = `https://you.owe.amanarya.com?ref=invite&email=${encodeURIComponent(record.invited_email)}`
```

Also update the plain text version of the email with the same URL.

**Note:** Android App Links (verified domain deep links) require `.well-known/assetlinks.json` on the domain, which is a separate infrastructure task. For now, we use the web URL as the primary CTA — the web landing page will have an "Open in App" button that uses the `owe://` scheme.
</task>

<task id="15-03-B" wave="2">
### B. Web landing page

This is a static/semi-static page deployed at `you.owe.amanarya.com`. Since the Owe app has a web build (Expo Router web), the landing page can be:

**Option 1 (Recommended):** A dedicated route in the existing Expo web app at the root — `app/index.tsx` already serves as the dashboard. We can add a route or redirect logic.

**Option 2:** A separate static HTML page deployed independently.

**Recommended approach for MVP:** Update the email CTA to point to the web app URL (the Expo web build). When the user arrives:
- If authenticated → redirect to `/invites` (pending invites screen)
- If not authenticated → show sign-in/sign-up screen, then redirect to `/invites` after auth

Since Expo Router already handles this flow (unauthenticated users see `(auth)` stack, authenticated see `(app)` stack), the email CTA just needs to point to the web app with a query param or route that resolves to the invites screen.

**Implementation:**
1. Set the email CTA to `https://you.owe.amanarya.com/invites`
2. After auth, the user lands in the `(app)` stack and navigates to invites
3. The `useClaimInvites` hook from Plan 15-02 auto-claims any pending invites

**For mobile users without the app:**
- The web page works as a PWA-like experience
- Add a "Download on Play Store" button/banner to the invites page (when we have a Play Store listing)
</task>

<task id="15-03-C" wave="2">
### C. Deep link handling for native app

If the user has the app installed and clicks a link with `owe://invites`:

1. Expo Router automatically resolves `owe://invites` to the `/invites` route
2. No additional deep link handler code needed — Expo Router's URL-based routing handles this

**For Android App Links (future enhancement):**
- Configure `intentFilters` in `app.json` for `you.owe.amanarya.com`
- Deploy `.well-known/assetlinks.json` on the domain
- This would make `https://you.owe.amanarya.com/invites` open directly in the app

For MVP, the web-first approach (Option B above) is sufficient.
</task>
